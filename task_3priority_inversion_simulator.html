OS Theory · Why this matters

Priority Inversion occurs when a high-priority task is indirectly blocked by a lower-priority task because of shared mutex contention. The classical 3-task scenario:

Task L (low) holds a mutex. Task H (high) wakes up and requests the same mutex, so it goes BLOCKED. So far this is correct. The bug appears when Task M (medium, doesn't need the mutex) becomes ready: M preempts L (because M > L by priority). Now H is effectively waiting for M to finish — even though H has higher priority than M. The priority order is inverted.

Mars Pathfinder (1997) hit exactly this bug. The high-priority bus management thread blocked on a mutex held by a low-priority meteorological task. Medium-priority communications work kept preempting the meteorological task. The bus thread's watchdog timer eventually expired and the system rebooted itself repeatedly. JPL fixed it by enabling Priority Inheritance on the VxWorks mutex (the flag was already there — they just had it disabled).

Priority Inheritance Protocol: when a higher-priority task blocks on a mutex, the current holder temporarily inherits the blocker's priority. This way, no medium-priority task can preempt the holder during its critical section. Once the mutex is released, the original priority is restored.
