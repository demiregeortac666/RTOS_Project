OS Theory · Why this matters

Deadlock is a permanent freeze that occurs when a set of tasks each hold a resource the other needs and none can proceed. Coffman's four necessary conditions (all must hold simultaneously):

  • Mutual exclusion: at least one resource (mutex) is held in non-shareable mode.
  • Hold and wait: a task holding a resource is allowed to request more.
  • No preemption: a held resource can only be released voluntarily by the task.
  • Circular wait: a cycle exists in the resource-allocation graph (Task A waits for Task B's resource, Task B waits for Task A's resource).

In our scenario, Task A grabs M_Sensor first, then needs M_Disk. Task B grabs M_Disk first, then needs M_Sensor. The reverse acquisition order is what creates the cycle.

Mutex Timeout (this fix): instead of blocking forever on acquire(), the call returns failure after a fixed time (4 ticks here). The task then performs rollback: it releases every resource it currently holds, backs off briefly, and retries from the start of its critical section. This breaks the no preemption condition: a held resource can now be lost (forced release), so the cycle cannot persist. POSIX provides this as pthread_mutex_timedlock(); FreeRTOS provides it as the xTicksToWait parameter on xSemaphoreTake().

Alternative fix — Priority Ceiling / Lock Ordering: require all tasks to acquire mutexes in the same global order (e.g. always M_Sensor before M_Disk). This breaks the circular wait condition by construction. It is preferred when feasible because it has zero runtime overhead, but it requires global agreement at design time.
